# 오픈소스 통합 기준

## 결론

이 프로젝트는 통째 포크보다 기능 단위 통합을 우선한다.

사업계획서 코워크 에이전트의 핵심은 쉬운 화면, 공고문 해석, 좋은 사례 활용, 사용자의 아이디어 보존이다. 큰 오픈소스 프로젝트를 그대로 합치면 기능은 늘 수 있지만 사용 난이도와 운영 부담이 함께 커진다.

## 1순위 통합

### Mermaid

- 목적: 논리모형, 추진 흐름, 성과 연결 구조도 생성
- 방식: 브라우저에서 바로 렌더링
- 현재 상태: `시각자료` 탭에 적용 완료
- 이유: 서버가 필요 없고, Markdown과 GitHub에서도 읽히며, 사업계획서 초안에 그대로 첨부하기 쉽다.

### 문서 텍스트 추출

- 목적: PDF, DOCX, HWPX 공고문과 사례 문서 읽기
- 후보: PDF.js, Mammoth.js, Docling
- 적용 순서:
  1. PDF.js로 PDF 텍스트 추출
  2. Mammoth.js로 DOCX 텍스트 추출
  3. 복잡한 표와 스캔 문서는 별도 로컬 도구 또는 외부 AI 질문으로 처리

### 사례 검색 고도화

- 목적: 좋은 사업계획서 약 30개에서 현재 공고와 맞는 문단 찾기
- 후보: MiniSearch, FlexSearch, Transformers.js
- 적용 순서:
  1. 현재 키워드 검색 유지
  2. MiniSearch로 빠른 한글/키워드 검색 개선
  3. 필요 시 로컬 임베딩 또는 서버형 벡터 검색으로 확장

## 참고할 큰 프로젝트

### OpenDocuments

- 역할: 로컬/Google Drive/문서 RAG/MCP 구조 참고
- 장점: 문서 검색, 인용, Google Drive, MCP, 로컬 실행 방향이 우리 목적과 가깝다.
- 주의: 그대로 합치면 서버형 구조가 되어 사용자가 어려워질 수 있다.
- 권장: 별도 브랜치에서 실행해 보고, 필요한 구조만 가져온다.

### GrantKit

- 역할: 그랜트 작성 도구의 파일 구조와 검증기 아이디어 참고
- 장점: grant writing 맥락이 우리 프로젝트와 가깝다.
- 주의: 미국 연구비 맥락이므로 한국 지원사업 양식에는 바로 맞지 않는다.
- 권장: 문서 구조, 검증 체크리스트, 에이전트 질문 흐름만 참고한다.

### Excalidraw / tldraw

- 역할: Mermaid로 만든 구조도를 사람이 편집 가능한 그림으로 발전
- 장점: PNG/SVG 내보내기와 시각적 편집에 강하다.
- 주의: 앱이 무거워질 수 있으므로 바로 내장하지 않는다.
- 권장: Mermaid를 먼저 쓰고, 이후 필요하면 별도 편집 탭으로 붙인다.

## 머지 전 체크리스트

- 라이선스가 MIT, Apache-2.0, BSD 계열인가?
- GitHub Pages 정적 앱에서 바로 작동하는가?
- 사용자가 새 계정, 서버, Docker를 몰라도 되는가?
- 한국어 공고문과 사업계획서 양식에 맞게 바꿀 수 있는가?
- 원본 문서와 개인정보가 외부 서버로 자동 전송되지 않는가?
- 기능이 사용자의 작업 시간을 실제로 줄이는가?

## 현재 방향

1. 앱은 단순하게 유지한다.
2. 브라우저에서 가능한 기능부터 붙인다.
3. Google Drive는 원본 보관소로 사용한다.
4. 로컬/브라우저 저장은 빠른 작업실로 사용한다.
5. 무료로 어려운 판단은 AI 질문 생성 흐름으로 처리한다.
6. 서버형 RAG는 필요성이 충분해졌을 때 별도 모듈로 붙인다.
